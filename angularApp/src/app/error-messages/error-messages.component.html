Error Messages Best Practices

Bad pattern — inline errors repeated per field

Inline error logic is scattered across every field. Changing the message or condition means editing every template where that field appears.

@if (emailDemo1.invalid && emailDemo1.touched) { @if (emailDemo1.hasError('required')) { This field is required } @if (emailDemo1.hasError('email')) { Enter a valid email address } }
@if (passwordDemo1.invalid && passwordDemo1.touched) { @if (passwordDemo1.hasError('required')) { This field is required } }

Playground

{{ formDemo1.value | json }}
Status: {{ formDemo1.status }}

Good pattern — centralized error component

The same form, the same validators — but error display is delegated to <app-error-message>. Messages live in one place and the template stays lean.

Playground

{{ formDemo2.value | json }}
Status: {{ formDemo2.status }}

Controlling when errors appear: showOn

The same control, three different trigger strategies. Type or leave the field to see each mode activate at a different moment.

touched — shows after the user visits the field
dirty — shows after the user changes the value
always — shows immediately on load
